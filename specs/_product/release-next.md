# Release: Tome Subsystem

## Goal
- Ship Tome, a manual post-merge documentation curator for Starlight docs sites that classifies each commit (or branch-level diff) against the four Diátaxis quadrants and runs only the writer skills the classifier selects.
- v1 introduces seven prompt-only skills under `src/deviate/prompts/skills/tome-*/`; the DeviaTDD repo's Python 3.13 + Typer stack is preserved without modification.
- v1.x (this iteration) adds the information-architecture (IA) contract: classifier-emitted `layer_order` / `parent` / `next` / `group` columns, writer-emitted `prev` / `next` frontmatter, per-quadrant theme sub-directories pre-created by setup, per-quadrant `<quadrant>/index.md` landing pages, per-quadrant `<quadrant>/_meta.yml` and per-theme `<quadrant>/<theme>/_meta.yml` ordering files, writer-owned content for `<quadrant>/index.md` and per-theme `_meta.yml` `pages:` appends, the `index-first` ordering invariant enforced by `[FAIL-INDEX-FIRST]`, and per-writer length budgets enforced by the verifier.

## Constraints
- Prompt-only in v1: each Tome skill lives as a static `SKILL.md` under `src/deviate/prompts/skills/tome-*/`. Python modules, Typer sub-apps, and runtime code remain out of scope for this iteration.
- The CLI surface (`deviate tome <phase>`) is deferred to a future iteration; v1 is prompt-only and gathers usage data first (per `specs/_product/architecture.md:21`).
- `FLOW-09` (Tome Verify Docs) emits a human-readable report. Verifier outputs are not auto-routed back to writers; humans drive the verification loop (per `specs/_product/architecture.md:20`).
- Each skill inlines the schemas it needs. v1 ships without a shared contracts module (`tome/contracts.py` or equivalent) (per `specs/_product/architecture.md:19`).
- Writer quadrant discipline: each of `tome-write-tutorial`, `tome-write-how-to`, `tome-write-reference`, `tome-write-explanation` is confined to its assigned quadrant. Within its quadrant, a writer honors the classifier's `target_file` path verbatim — if the path includes a theme sub-dir (e.g., `how-to/tdd-micro-cycle/red.md`), the writer writes under it. Writers that need to touch a different quadrant flag back to `tome-classify` for re-classification (per `specs/_product/architecture.md:64`).
- Tome output is decoupled from DeviaTDD's runtime stack. Skill outputs (Starlight sites, `apps/docs/`, `content.config.ts`, `.astro` files) live in target repos that consume the skills, not in the DeviaTDD repo (per `specs/_product/architecture.md:18`).
- Frontmatter schema parity: the field set declared in `apps/docs/src/content.config.ts` agrees field-for-field with the inline schema in the C2-C5 prompts. Drift between the two surfaces as a `FLOW-09` finding (per `specs/_product/architecture.md:149`). The schema is nine fields; the two new in this iteration are `prev` and `next`.
- The DeviaTDD repo keeps its Python 3.13 + Typer stack; new skills ship as static `SKILL.md` files under `src/deviate/prompts/skills/tome-*/` and introduce no `package.json`, `astro.config.mjs`, or Node toolchain (per `specs/_product/architecture.md:17` and `:212`).

## Acceptance Criteria

- `tome-setup` scaffolds `apps/docs/` with the four quadrant directories (`tutorials/`, `how-to/`, `reference/`, `explanation/`), per-quadrant `<quadrant>/index.md` (the navigation pivot, at the canonical Starlight path), per-quadrant `<quadrant>/_meta.yml` (the sidebar manifest at the canonical Starlight path, with `pages: [index.md, ...]` to enforce the `index-first` ordering invariant), the per-theme sub-directories under each quadrant (e.g., `how-to/tdd-micro-cycle/`, `reference/cli/`, `explanation/architecture/`), the per-theme `<quadrant>/<theme>/_meta.yml` ordering files, the root `index.md` (rewritten with intro + `## Quadrants` section linking to each `<quadrant>/index.md`), and `src/content.config.ts` extending `docsSchema()` with the nine Tome frontmatter fields (`title`, `description`, `doc_type`, `status`, `last_verif...
- `tome-setup`'s optional starter set (one architecture explanation, one config reference, one first-task how-to, one first-run tutorial) seeds by default and is suppressible via an opt-out flag; the starter set lives under theme sub-dirs (e.g., `how-to/getting-started/starter-first-task.md`) (per `specs/_product/flows/flows-tome.md:267` and the per-quadrant theme mapping in `specs/_product/domain-model.md` §ThemeGroup).
- `tome-classify` in default `HEAD~1` mode emits a classification report whose capability table contains the columns `capability`, `evidence`, `audience`, `doc_type`, `action`, `target_file`, `confidence`, **`layer_order`**, **`parent`**, **`next`**, **`group`** and whose `action` column is restricted to the enum `create | update | no-change | human-review | setup-required`; the IA columns are derived deterministically from the capability name using the canonical phase order and the ThemeGroup mapping in `specs/_product/domain-model.md` §ThemeGroup (per `specs/_product/flows/flows-tome.md:22-28`, `specs/_product/architecture.md:46-57`, and `specs/_product/domain-model.md` §Capability).
- `tome-classify` accepts `<sha>`, `--merge-base`, and `--working-tree` modes; `--merge-base` produces a branch-level classification scoped to the cumulative diff from `main`'s merge-base to `HEAD`; `--codebase` walks the entire repository without a diff and emits an exhaustive capability table (per `specs/_product/flows/flows-tome.md:12-13` and `:35`).
- `tome-classify` emits a row whose `action` is `setup-required` and halts without proposing target files when `apps/docs/` is absent; emits `no-change` when all changes in the diff are internal-only; emits `human-review` when the target file collides with an existing page in the wrong quadrant; emits `human-review` with `[NEW-THEME]` flagged when a capability does not map to any `ThemeGroup` (per `specs/_product/flows/flows-tome.md:31-34` and `specs/_product/architecture.md:50-57`).
- `tome-write-tutorial`, `tome-write-how-to`, `tome-write-reference`, and `tome-write-explanation` write exclusively to `apps/docs/src/content/docs/tutorials/`, `how-to/`, `reference/`, and `explanation/` respectively (root or theme sub-dir per the classifier's `target_file`); each writer rejects any target path outside its quadrant and surfaces a boundary violation (per `specs/_product/flows/flows-tome.md:73-76`, `:113`, `:151`, `:189` and `specs/_product/architecture.md:64-65`).
- Every file emitted by a writer carries valid Tome frontmatter with the nine fields `title`, `description`, `doc_type`, `status`, `last_verified_at`, `verified_sha`, `related_issues`, `prev`, `next`; the `doc_type` value matches the quadrant directory the file lives in (e.g., `tutorial` only inside `tutorials/`); the `prev` / `next` values are derived from the classifier's `parent` / `next` columns and are `null` only for the per-quadrant `<quadrant>/index.md`, the first page in a theme, or the last page in a theme (per `spec...
- Every writer self-enforces a length budget: how-to ≤ 80 lines, tutorial ≤ 120 lines, explanation ≤ 90 lines; reference = tables dominate with no narrative paragraph longer than 2 sentences. Drift beyond budget surfaces as a `[FAIL-LENGTH]` finding in the verifier (per `specs/_product/flows/flows-tome.md:73-76`, `:113`, `:151`, `:189` and `specs/_product/architecture.md:70` and `:101`).
- The field set in `apps/docs/src/content.config.ts` agrees field-for-field with the inline Tome frontmatter schema in the C2-C5 prompts (nine fields); any drift between the two is flagged by `tome-verify-docs` as a finding (per `specs/_product/architecture.md:150` and `specs/_product/flows/flows-tome.md:281`).
- `tome-verify-docs` runs after at least one of `FLOW-05..FLOW-08` has produced a file; emits a report with PASS items, FAIL items, boundary violations, IA reachability findings (`[FAIL-IA]`), length budget findings (`[FAIL-LENGTH]`), and recommended files to commit; does not auto-route failures back to writers (per `specs/_product/flows/flows-tome.md:218-238` and `specs/_product/architecture.md:96-105`).
- `tome-verify-docs` IA reachability check: every updated file under a quadrant MUST appear in that quadrant's per-theme `<quadrant>/<theme>/_meta.yml` ordering (or in the per-quadrant `<quadrant>/_meta.yml` for files at the quadrant root); the per-quadrant `<quadrant>/_meta.yml` MUST list `index.md` as its first `pages:` entry (the `index-first` ordering invariant; drift surfaces as `[FAIL-INDEX-FIRST]`); every content page MUST carry `prev` / `next` frontmatter (or `prev: null` if it is the first page in its theme; or `next: null` if it is the last). The per-quadrant `<quadrant>/index.md` and the root `index.md` are exempt from the `prev` / `next` check (navigation pivots) (per `specs/_product/architecture.md:100` and `specs/_product/flows/flows-tome.md:228-233`).
- All seven Tome skills run as LLM-mediated prompts; the DeviaTDD repo's Python 3.13 + Typer stack is preserved and no `package.json`, `astro.config.mjs`, or Node toolchain is added (per `specs/_product/architecture.md:17-21` and `:212`).
